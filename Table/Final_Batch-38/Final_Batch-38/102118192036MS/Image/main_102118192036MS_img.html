In order to arbitrarily select possible locations in the reservoir, the wells are placed as in this spacing (11 × 11 = 121 wells as shown in Fig. 1[OD2]). These well locations will be used for further evaluation of their potential for selection as candidates for a given number of required wells.

For this step, each well is simulated through a streamline simulator (in this study, FRONTSIM). Each producer is set to produce with a BHP limit to illustrate their maximum flow potential. The results from this step are (i) flow potential from each well and (ii) streamlines traced from each producer to the boundary (Fig. 2).

The objective of this step is to map data from the streamlines to each simulation grid for further calculations. Generally, the generated streamlines data is in line format, which is stored in .SLN files. One single grid may be penetrated by more than one line. The MATLAB binary reader is used to extract information from the .SLN file and to calculate the average time-of-flight from the line to each grid. The result can accurately mimic the streamline time-of-flight result to the grid (Fig. 3).

The HCPV is then weighted by a time-of-flight value. A low time-of-flight near the wellbore will have a high weight whereas a farther distance with an increased particle travel time will have a lower weight. We define the relationship between a normalized time-of-flight weight versus a time-of-flight in each grid cell as a linear relationship with a cut-off value (Fig. 4 and Fig. 5). The largest weight will be given at time-of-flight = 0 (near the well bore), the value beyond cut-off will be treated as 0. The cut-off time-of-flight is defined as 30 years.

Instead of running reservoir simulations, we perform a proxy calculation of the drainage volume by turning the effective HC drainage volume into a mathematical matrix union with a given number of wells, and then search for the case that provides the largest score. The calculation of a matrix union can be done at an extremely fast speed, less than 0.1 second per trial, which make the exhaustive search possible. Additionally, a parallel computing technique can also be implemented into this workflow to further expedite the computational efficiency. The overall exhaustive optimisation workflow is illustrated in Fig. 7.

The algorithm herein has been conceptually tested with 2D synthetic homogeneous reservoir simulation examples. The intention of this test is to observe what the spacing derived from the algorithm looks like when applied to the 2D problem that humans can visualize. The algorithm has been applied to search optimal locations from 1 well to 5 wells (Fig. 8).

The work plan to deliver the application package to perform the workflow in this study is called "Smart Production Well Drainage Optimizer (SMARTDRAIN)". The workflow from running streamline simulator, digitizing streamlines, calculating efficient drainage volume and optimizing by exhaustive search are included in the package using the MATLAB compiler. In order to ensure greater specific user friendliness, engineers need to prepare input only for the data in the provided MS Excel template. The software provides results in MS Excel, with the program structure illustrated in Fig. 9.

"Field X," which is an offshore gas-condensate field in its preliminary development phase, is simulated as a reference model in this study. The reservoir model contains around 3 million cells with 600K active cells. The development plan aims to find proper locations to maximize plateau duration. A total of 23 wells representing locations away from GWC has been initially planned, spreading around the reservoir (Fig. 10). The SMARTDRAIN package has been tested with this field scale reservoir simulation model to locate optimal locations for the field development plan.

With heterogeneity, the drainage area is not circular as in the homogeneous case, but the drainage is extended toward the preferred flow-path with a high permeability trend in a NE-SW direction (Fig. 13). Additionally, the high permeability regions are primarily located in southern part of the reservoir. This information suggests the majority of development wells should focus on the southern region more than the northern region. The proper number of wells in each region is the primary requirement for this study where the SMARTDRAIN serves as the key engine to deliver the answer.

Two well placement development scenarios are compared. The first scenario uses a manual human justification to place 12 producers throughout the reservoir (Fig. 14). The second scenario uses the developed SMARTDRAIN algorithm in this work to locate the 12 producers (Fig, 15). The first scenario tends to place the wells with equal spacing more than the latter scenario where the wells are spaced more densely in regions with high HCPV.

The primary requirement of Field × development is a plateau production period. The result from the two scenarios, manual justification and optimisation by SMARTDRAIN, are compared in Fig. 16. It can be observed that the well placement suggested by SMARTDRAIN has improved the plateau duration by 5.1% more than from manual justification. Despite various attempts from manual justification to locate well placement on scenarios with proper well spacing size, an improvement from this algorithm-assisted workflow has been clearly shown to occur. This improvement also proves the necessity of integrated reservoir simulations to improve decision-making on selections for well placement.
